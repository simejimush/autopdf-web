# AutoPDF Quality Rules (必須版)

このプロジェクトは「Gmail → PDF生成 → Google Drive保存」を自動化するSaaS。
品質の最優先は **安全性 / 監査可能性 / 保守性**。次に **正確性 / ユーザー体験 / コスト**。

---

## 0. 最重要方針（迷ったらここ）

- “とりあえず動く”より **壊れにくい** を優先する
- **ユーザーデータ漏えいに繋がる可能性**が1%でもある実装はしない
- 仕様が曖昧なら、推測で作り込まず「安全側の仮実装（ガード・ログ・失敗）」に寄せる

---

## 1. 認証・認可（Supabase前提）

- すべてのユーザーデータは `user_id`（= auth.uid()）に必ず紐づける
- クライアント/Server ClientでアクセスするDBは **RLS前提**。RLSを回避する設計はしない
- `supabaseAdmin` は **最小範囲**・**API内部のみ**・**ユーザー境界を破らない用途のみ**
- “ログインしていない”場合は必ず `401` を返す（曖昧なエラーにしない）

**必須チェック**

- DBテーブルに `user_id` が無いなら、原則として追加する（例外は公開マスタのみ）
- APIは「呼び出しユーザーがそのrule/runを所有しているか」を必ず担保する

---

## 2. 入力検証（外部入力はすべて疑う）

対象: query params / body / headers / pathname / gmail_query / folderId / ruleId など

- 形式チェック（空・長さ・型・UUID・許可文字）を行う
- そのままDBに突っ込まない。必ずバリデーションしてから
- エラーは `400`（入力不正）として返す。サーバの例外にしない

---

## 3. Cron / バックグラウンド実行（公開API事故を防ぐ）

- Cronエンドポイントは **必ず secret（CRON_SECRET 等）で保護**
- secretが無い / 不一致なら `401` or `403`
- Cronは「処理を回す」だけにして、実処理は共通関数（手動Runと共有）へ寄せる
- Cronの実行結果は必ずDB（runs）に残し、後から追えるようにする

---

## 4. 外部API（Google OAuth / Gmail / Drive）は失敗前提

- Google APIは必ず失敗する前提で設計する（トークン失効・権限不足・rate limit・一時障害）
- 失敗時は **握りつぶさず**、ユーザーに分かる形で `runs.status=error` と `error_code` を記録
- “再実行して安全”な作り（冪等性）を優先する

**トークン/秘密情報**

- access_token / refresh_token / client_secret をログに出さない
- 暗号化・復号の責務を分離し、呼び出し側に平文をばら撒かない

---

## 5. 監査ログ（runsは生命線）

- 重要な処理は必ず `runs` に記録する（開始/終了/ステータス/件数/エラー）
- 失敗したら「どこで」「なぜ」が分かる `error_code` と短い `message` を残す
- `message` はユーザー向け/運用向けに簡潔に（機密情報は入れない）
- 例外は catch して `runs` に落としてから返す（ただし隠蔽しない）

推奨エラー分類例（増やしてOK）:

- AUTH_REQUIRED / FORBIDDEN
- GOOGLE_TOKEN_INVALID / GOOGLE_PERMISSION_DENIED
- GMAIL_QUERY_INVALID / DRIVE_FOLDER_INVALID
- RATE_LIMIT / TEMPORARY_UNAVAILABLE
- DB_RLS_DENIED / DB_CONSTRAINT

---

## 6. 冪等性（同じメールを何度処理しても壊れない）

- “同じ入力が2回来ても同じ結果になる”設計を優先
- processed_emails 等で「処理済み判定」を行い、重複生成・重複アップロードを防ぐ
- 途中で落ちても、再実行で回復できるようにする（途中状態を残さない/残すなら整合性を守る）

---

## 7. パフォーマンス（runs増加でUIが重くならない）

- 一覧API/UIは **必要なフィールドだけ**を取る（SELECT \* しない）
- runsは増える前提。N+1を避け、limit/order/indexを意識する
- 「最新1件」などはDB側で取る（クライアントで全件処理しない）

---

## 8. コード設計（重複禁止・責務分離）

- “同じロジックを2箇所に書かない”（手動Run / Cron / UI計算の重複を避ける）
- 実処理は `lib/` の関数に寄せ、Routeは薄く保つ
- UI表示用の整形は「UI層」へ。APIはデータを正しく返すことに集中

---

## 9. 変更のしかた（安全に壊さない）

- 変更は「小さく」「局所的に」「既存挙動を壊さない」順で
- 既存の成功パス/失敗パスに影響する場合、必ずログとruns記録を確認できるようにする
- “なんとなく動いた”で終わらせず、再現手順と期待結果を言語化する

---

## 10. 出力フォーマット（AIへの要求）

コード提案時は必ず以下を含める:

- 変更ファイルのフルパス
- 差分の範囲（置換なら「どこからどこまで」）
- 追加したガード/ログ/runs更新ポイントの説明（短く）
- 失敗時の挙動（どのstatus/error_codeになるか）

以上。
