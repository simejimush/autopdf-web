# feature-dev テンプレート

このテンプレートは、機能開発を「設計→実装→レビュー」の順で安全に進めるためのもの。

---

## 🧠 基本ルール

- いきなり実装から始めない
- 必ず「要件 → 設計 → 実装 → レビュー」の順で進める
- 既存コードを最大限再利用する
- シンプルな設計を優先する（過剰設計NG）

## ② 既存コード分析（explorer）

前提：

- Next.js App Router
- Supabase（RLSあり）
- rules / runs / processed_emails テーブルあり

分析すること：

- 関連API（例：/api/rules）
- DB構造（rulesテーブル）
- 再利用できる処理
- 既存の問題点

## ④ 実装

- ファイルパスを明示すること

- コピペ可能なコードを出すこと

---

## ⑤ レビュー（reviewer）

以下をチェック：

### バグ

- データ不整合ないか
- null / undefined対策

### セキュリティ

- RLSを突破していないか
- user_idの扱い問題ないか

### UX

- わかりにくい挙動ないか
- 誤操作防げているか

### 改善提案

- よりシンプルにできないか
- 将来的な拡張性

---

## 👥 役割分担（内部思考）

以下の3役で思考すること：

- explorer：
  既存コードを理解し、再利用可能な部分を特定する

- architect：
  最小コストで安全に実装できる設計を決める

- reviewer：
  本番運用で問題ないか厳しくチェックする

---

## ⚠️ 禁止事項

- いきなりコードを書き始める
- 既存コードを無視した設計
- 不要な抽象化・過剰設計
