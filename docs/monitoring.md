# Monitoring / Notifications（AutoPDF）

最終更新: 2026-04-19  
目的: AutoPDF の監視・通知運用を、実装と運用の両面でブレなく整理する。

---

## 1. 目的

AutoPDF は「一度設定したら、しばらく放置運用されやすい」性質がある。  
そのため、単にエラーを DB に残すだけでなく、**処理停止を早く検知し、管理者とユーザーの両方が気づける状態**を作ることを目的とする。

特に重視すること:

- エラーを握りつぶさず、必ず追跡できる形で残す
- 通知判定は message ではなく error_code ベースで行う
- 手動Run と Cron で監視ロジックを分けない
- 通知失敗が本処理全体を巻き込んで失敗しないようにする

---

## 2. 監視対象

監視対象は大きく3種類に分ける。

### 2.1 システムエラー

例:

- API 500
- DB書き込み失敗
- 想定外例外
- Cron 異常終了

対応:

- runs に error_code / message を記録
- 必要に応じて管理者通知
- 原因調査できるログを残す

### 2.2 ビジネスエラー

例:

- Gmail取得失敗
- Google Drive保存失敗
- Google接続切れ
- 保存先フォルダ不正
- Gmail検索条件不正

対応:

- runs に記録
- error_code に応じて管理者通知
- 再接続が必要なものはユーザー通知

### 2.3 ユーザー個別エラー

例:

- 特定ユーザーだけ Google 接続が切れている
- 特定ルールだけ失敗している
- 長期間成功していない

対応:

- /admin/errors や runs を使って追跡可能にする
- 必要に応じてユーザーへ再接続や設定確認を案内する

---

## 3. 監視の中心データ

監視の中心は `runs` とする。

最低限、以下を必ず記録する。

- status
- error_code
- message
- started_at
- finished_at

監視上の考え方として、**まず runs を見れば「何が、いつ、どのルールで失敗したか」が分かる状態**を維持する。

加えて、Google 接続状態の監視のために `google_connections` にも状態を持つ。

主な監視用カラム:

- last_success_at
- last_error_at
- last_error_code
- reauth_required
- last_user_notified_at
- last_user_notified_error_code

---

## 4. error_code ベースの判定方針

通知や監視の判定は、message ではなく `error_code` ベースで行う。

理由:

- message は文面変更で揺れやすい
- UI / Slack / メールで同じ判定基準を共有しやすい
- サポート・運用で切り分けしやすい

基本ルール:

- severity が `critical` または `high` のものは、原則として管理者通知対象
- `reauth_required = yes` のものは、原則としてユーザー通知対象
- 想定内ケース（例: `GMAIL_EMPTY_RESULT`）は通知しない
- 同一ユーザーへの重複通知は抑制する

主要な辞書と優先度は `docs/error-codes.md` を参照する。

---

## 5. 実行フローと監視の接点

監視ロジックは、手動Run と Cron で分けず、共通実行本体に寄せる。

基本フロー:

1. ルール実行開始
2. runs に開始記録
3. Gmail検索 / メール取得 / PDF生成 / Drive保存
4. 成功時は runs を success 更新
5. 失敗時は error_code / message を正規化して runs を error 更新
6. google_connections の監視状態を更新
7. 必要なら管理者通知 / ユーザー通知を呼ぶ
8. 通知失敗はログに残すが、本処理は失敗させない

この順序を守る理由:

- runs が監査の生命線だから
- 通知より先に、まず実行結果を確定させる必要があるから
- Slack / メール送信失敗が本来の PDF 処理失敗を上書きしてはいけないから

---

## 6. 管理者通知（Slack）

### 6.1 目的

管理者通知は、重大エラーや継続失敗を早く把握するために使う。

### 6.2 方針

- 通知先は Slack を優先
- 通知判定は error_code ベース
- 通知失敗時も本処理は止めない
- 最小限の監査情報を含める

### 6.3 通知内容の最小単位

- user_id
- rule_id
- error_code
- trigger
- 発生時刻

### 6.4 初期対象

初期は `critical` / `high` 相当のエラーを中心に通知する。  
詳細は `docs/error-codes.md` の `admin_notify` 方針に従う。

---

## 7. ユーザー通知（メール）

### 7.1 目的

ユーザーがサイトを開いていなくても、Google 再接続が必要な状態に気づけるようにする。

### 7.2 初期対象

初期のユーザー通知対象は以下の2つに限定する。

- `GOOGLE_TOKEN_INVALID`
- `GOOGLE_PERMISSION_DENIED`

理由:

- ユーザーが再接続や再同意をしない限り復旧しにくい
- 通知対象を広げすぎるとノイズが増える
- まずは Google 再接続系だけに絞ったほうが安全

### 7.3 送信条件

以下をすべて満たしたときだけ送る。

- runs.status = error
- error_code がユーザー通知対象
- google_connections.reauth_required = true
- 同一 user_id に24時間以内の通知がない

### 7.4 文面方針

件名:

- 【AutoPDF】Google連携の再接続が必要です

本文方針:

- 何が起きたか
- 何が止まっているか
- 何をしてほしいか
- 設定ページへの導線

初期段階では、以下は本文に含めない。

- 生の error_code
- user_id
- rule_id
- 技術的な詳細

### 7.5 実装責務

- `src/lib/monitoring/notifyUser.ts`
  - 重複通知抑制
  - メール文面生成
  - 送信成功後の通知状態更新

- `src/lib/email/sendEmail.ts`
  - 外部メールサービス呼び出し専用
  - to / subject / text を受けて送信
  - 失敗時は throw

---

## 8. 重複通知抑制

初期は `google_connections` に通知状態を持つ。

使用カラム:

- last_user_notified_at
- last_user_notified_error_code

初期ルール:

- 同一 user_id には24時間再送しない
- 送信成功時のみ通知時刻を更新する
- 本格的な通知履歴テーブルはまだ作らない

理由:

- 今必要なのは「通知しすぎ防止」であり、完全な通知監査台帳ではない
- Google 接続状態の監視は `google_connections` と相性がよい
- テーブル追加を最小にできる

---

## 9. /admin/errors の役割

`/admin/errors` は管理者向けのエラー一覧画面であり、最新の失敗を素早く把握するための UI とする。

役割:

- 最新 error ログを新しい順に確認する
- error_code / trigger / user_id で絞り込む
- user_id / rule_id / error_code / 発生時刻 を追えるようにする
- 監査・切り分けの入口にする

## 9.1 /admin/errors の管理者運用ルール

`/admin/errors` は、エラー調査の入口として使う。  
この画面だけで解決まで完結させるのではなく、**error_code を見て原因の種類を切り分け、影響範囲を判断し、次の確認先を決める**ための画面とする。

### 基本の見方

管理者はまず以下をこの順で見る。

1. `error_code`
2. `user_id`
3. `rule_id`
4. `message`
5. `trigger`
6. 発生時刻

最初に判断することは以下。

- 単発エラーか
- 同じ user_id に集中しているか
- 同じ rule_id だけで起きているか
- 複数ユーザーで同時多発しているか

### 基本フロー

1. `error_code` を確認する
2. 同じ `user_id` に集中しているか確認する
3. 同じ `rule_id` に限定されるか、複数 `rule_id` に広がるか確認する
4. error_code の種類に応じて対処方針を決める
5. 必要に応じてユーザーへ案内、または管理者側で設定確認を行う
6. 修正後は **1件だけ手動 Run** して success に戻るか確認する

### error_code ごとの基本行動

#### 1. Google 再接続系

対象例:

- `GOOGLE_TOKEN_INVALID`
- `GOOGLE_PERMISSION_DENIED`

行動:

- 同じ `user_id` で複数 `rule_id` に出ていないか確認する
- 複数ルールで同じユーザーに出ている場合は、ルール個別不具合ではなく Google 接続切れと判断する
- ルール設定は触らず、再接続案内を優先する
- ユーザー通知メールが送られている前提で、必要なら追加で再接続を促す
- 再接続後に 1 ルールだけ手動 Run して復旧確認する

#### 2. ルール設定不備系

対象例:

- `GMAIL_QUERY_INVALID`

行動:

- 該当 `rule_id` を特定する
- ルールの Gmail 検索条件を確認する
- 必要に応じて検索条件を修正する、またはユーザーへ修正を案内する
- 修正後に手動 Run で確認する

#### 3. Drive 保存先系

対象例:

- `DRIVE_FOLDER_INVALID`
- `DRIVE_PERMISSION_DENIED`

行動:

- 該当 `rule_id` の Drive 保存先を確認する
- フォルダ削除、フォルダID誤り、権限不足を疑う
- 必要に応じてユーザーへ保存先再設定を案内する
- 修正後に手動 Run で確認する

#### 4. 一時障害系

対象例:

- `RATE_LIMIT`
- `TEMPORARY_UNAVAILABLE`
- `TIMEOUT`

行動:

- 単発か連続かを確認する
- 複数ユーザー・複数ルールで同時発生していないか確認する
- 単発なら即修正せず様子見とする
- 継続する場合は Slack 通知や Vercel Logs を確認する
- 復旧後に必要なら再実行する

#### 5. 内部調査優先系

対象例:

- `DB_RLS_DENIED`
- `DB_CONSTRAINT`
- `DB_INSERT_FAILED`
- `DB_UPDATE_FAILED`
- `UNKNOWN`

行動:

- ユーザー側設定は先に触らない
- Slack 通知、Vercel Logs、直近デプロイ差分を確認する
- 必要に応じて開発対応を優先する
- ユーザー操作で解決する前提で案内しない

### 判断の目安

- **同じ user_id × 複数 rule_id**  
  → ユーザー単位の問題の可能性が高い  
  例: Google 接続切れ

- **同じ rule_id のみ**  
  → ルール設定や保存先の問題の可能性が高い

- **複数 user_id で同時多発**  
  → 外部API障害、実装不具合、共通設定不備の可能性が高い

### 運用上の注意

- `/admin/errors` は管理者専用画面であり、見た目より情報の正確性を優先する
- 調査の中心は `runs` とログであり、この画面は入口とする
- message ではなく `error_code` を基準に判断する
- 修正後の確認は、いきなり全件ではなく **1件の手動 Run** を優先する
- Google 再接続系は、ルール個別修正より再接続確認を優先する

注意:

- この画面は管理者専用
- 一般ユーザー向けの見た目より、情報の正確性を優先する
- 詳細調査の中心は runs とログであり、この画面は入口とする

## 9.2 /admin/errors の管理者向け簡易版

### まず見るもの

1. `error_code`
2. `user_id`
3. `rule_id`

### 基本判断

- **同じ user_id × 複数 rule_id**
  - ユーザー単位の問題の可能性が高い
  - 例: Google 接続切れ

- **同じ rule_id のみ**
  - ルール設定や保存先の問題の可能性が高い

- **複数 user_id で同時多発**
  - 外部API障害、実装不具合、共通設定不備の可能性が高い

### error_code ごとの基本行動

- `GOOGLE_TOKEN_INVALID` / `GOOGLE_PERMISSION_DENIED`
  - Google 再接続を確認
  - ルール個別修正はしない
  - 再接続後に 1 件だけ手動 Run で確認

- `GMAIL_QUERY_INVALID`
  - ルールの Gmail 検索条件を確認
  - 修正後に手動 Run で確認

- `DRIVE_FOLDER_INVALID` / `DRIVE_PERMISSION_DENIED`
  - 保存先フォルダ設定や権限を確認
  - 修正後に手動 Run で確認

- `RATE_LIMIT` / `TEMPORARY_UNAVAILABLE` / `TIMEOUT`
  - 単発なら様子見
  - 継続するなら Slack / Logs を確認

- `DB_RLS_DENIED` / `DB_*` / `UNKNOWN`
  - ユーザー設定は触らず、先に開発調査

### 運用ルール

- 判断は `message` ではなく `error_code` ベースで行う
- 修正後はいきなり全件再実行せず、**1件だけ手動 Run** で確認する
- `/admin/errors` は調査の入口であり、詳細確認は `runs` とログで行う

## 9.3 /admin/errors のアクセス制御メモ

現行の `/admin/errors` は、**特定の管理者メールアドレスをコード内で固定判定する方式**で保護している。

### 現在の方式

- 未ログインユーザーは `/login` へリダイレクト
- ログイン済みでも、管理者メールと一致しない場合は `/dashboard` へリダイレクト
- 一致した場合のみ `/admin/errors` を表示する

この方式は、**管理者が1名の間はシンプルで安全**に運用しやすい。  
一方で、将来的に管理者を増やす場合は拡張しづらい。

### 将来の拡張候補

管理者が複数になる、または権限管理を明確にしたくなったら、以下の方式へ移行を検討する。

#### 案1: `user_profiles` に管理者フラグを持つ

例:

- `is_admin: boolean`

方針:

- `/admin/errors` 表示可否を email 固定ではなく `is_admin = true` で判定する
- 管理者追加時にコード変更ではなく DB 更新で対応できる

#### 案2: 専用の管理者テーブルを持つ

例:

- `admin_users`
  - `user_id`
  - `role`
  - `created_at`

方針:

- `user_profiles` とは分離して権限管理だけ独立させる
- 将来的に権限レベル（viewer / operator / owner など）を増やしやすい

### 現時点の判断

現時点では、管理者が限定されているため **メール固定判定のままで問題ない**。  
ただし、以下のどちらかが発生した時点で見直す。

- 管理者を複数人にしたい
- `/admin/errors` 以外にも管理者専用画面が増える

### 注意

移行時は、単に UI を隠すのではなく、**サーバー側でアクセス制御を判定すること**を前提とする。  
管理画面は必ず「未ログイン不可」「非管理者不可」をサーバー側で保証する。

## 10. ローンチ前の暫定事項

現時点では、ユーザー通知メールは暫定運用が含まれる。

暫定事項:

- `EMAIL_FROM` はテスト用設定
- Resend 無料テスト制限の都合で、送信先を一時固定して確認している
- 独自ドメイン未認証のため、本番向けメール送信設定は未完了

このため、**現時点のユーザー通知は「配線と実メール受信確認は完了、ただし送信設定は暫定」** と整理する。

## 11. 後で戻すもの

独自ドメイン取得・メールドメイン認証後に、以下を戻す。

- notifyUser.ts の送信先固定を元に戻す
- `EMAIL_FROM` を正式な差出人アドレスに変更する
- 本番宛先で再通知テストを行う
- 必要なら迷惑メール対策を再確認する

## 13. 設計原則まとめ

- エラーは握りつぶさず、必ず runs に記録する
- 通知判定は error_code ベースで行う
- 手動Run と Cron で監視ロジックを分けない
- 通知処理は Route に直書きせず、共通関数に寄せる
- 通知失敗が本処理全体を巻き込まないようにする
- Google 接続切れ系は reauth_required を判定できる形で扱う
- 監視は「OAuth状態」だけでなく「最後に正常保存できたか」も重視する
